## 二、架构设计评审

### ✅ 优秀设计

| 设计点           | 文件                | 评价                                       |
| ------------- | ----------------- | ---------------------------------------- |
| **状态机模式**     | `boss_base.gd`    | 5 状态 FSM（ENTER/IDLE/ATTACK/TRANSFORM/DYING）设计清晰，每个状态用独立内部类实现，职责分离 |
| **动态 Hitbox** | `boss_base.gd`    | `_create_hitbox()` 动态创建 Area2D 子节点解决 CharacterBody2D 无 body_entered 信号问题，设计巧妙 |
| **对象池动态扩容**   | `pool_manager.gd` | `auto_expand_limit` + `expand_increment` 机制解决密集波次池满问题，兼顾性能与灵活性 |
| **JSON 配置覆盖** | `boss_base.gd`    | 运行时从 JSON 加载配置覆盖导出值，支持快速调参，无需修改场景文件      |
| **软引用模式**     | `boss_base.gd`    | `$X if has_node("X") else null` 避免节点不存在导致 _ready 中断，健壮性好 |

### ⚠️ 架构改进建议

#### 1. BossBase 状态机未加入场景树

**位置**: `boss_base.gd:67-68`
```gdscript
var _state_machine: StateMachine = null
```

**问题**: `_init_state_machine()` 中 `_state_machine = StateMachine.new()` 创建后未 `add_child`，虽然当前通过手动调用 `_state_machine.current_state().update(delta)` 工作，但违背了 Godot 节点生命周期规范。StateMachine 的 `_process` 不会被自动调用，依赖外部手动驱动。

**建议**: 改为 `add_child(_state_machine)`，让状态机通过自身 `_process` 驱动，减少耦合。

#### 2. EnemyBase.object_pool 字段未使用

**位置**: `enemy_base.gd:35`
```gdscript
var object_pool: ObjectPool = null
```

**问题**: 该字段类型为 `ObjectPool`（scripts/object_pool.gd），但实际对象池操作已通过 `PoolManager` 单例完成。`_return_to_pool()` 中检查了 `object_pool != null`，但从无赋值路径，导致敌机始终走 `queue_free()`。

**建议**: 移除该字段或改为引用 `PoolManager`，保持接口一致性。

## 三、代码质量评审

### ✅ 代码质量亮点

- **注释完善**: 关键方法和变量都有清晰的注释说明用途和参数
- **错误处理**: 多处使用 `push_warning`/`push_error` 和 `has_method`/`has_signal` 安全检查
- **防御性编程**: 大量空值检查、边界检查
- **命名规范**: 变量/方法命名清晰，遵循 GDScript 约定（下划线前缀表示私有）

### ⚠️ 代码质量问题

#### 1. BossBase._process 未调用 super._process

**位置**: `boss_base.gd:187-192`
```gdscript
func _process(delta: float) -> void:
    if _state_machine != null and _state_machine.current_state() != null:
        _state_machine.current_state().update(delta)
    spiral_angle += delta * 120.0
```

**问题**: `extends EnemyBase` 但完全覆盖 `_process` 且不调用 `super._process()`，注释说明 BOSS 不需要父类的直线下移和弹幕逻辑，这是合理的，但 `EnemyBase._process` 中还有屏幕外检测逻辑（`global_position.y > viewport + 200` 时归还），BOSS 缺失此保护。

**建议**: 在 `_process_entry` 或 `_process_idle` 中添加屏幕外检测，防止 BOSS 意外飞出屏幕。

#### 2. LevelBase._get_enemy_scene_path 硬编码路径

**位置**: `level_base.gd:336-345`
```gdscript
func _get_enemy_scene_path(enemy_type: String) -> String:
    match enemy_type:
        "ki27_fighter":
            return "res://scenes/enemies/ki27_fighter.tscn"
```

**问题**: 这些场景文件实际不存在（所有敌机场景都映射到 `enemy_fighter.tscn`），但代码中硬编码了路径。当 SpawnManager 不可用时（`_spawn_enemies_direct` 分支）会加载失败。

**建议**: 与 SpawnManager 的 `_enemy_scene_map` 保持一致，或者直接委托给 SpawnManager 获取路径。

#### 3. SpawnManager._enemy_scene_map BOSS 映射错误

**位置**: `spawn_manager.gd:160-162`
```gdscript
_enemy_scene_map["BOSS_bomber"] = placeholder
_enemy_scene_map["BOSS_cruiser"] = placeholder
_enemy_scene_map["BOSS_fortress"] = placeholder
```

**问题**: BOSS 类型映射到了 `enemy_fighter.tscn` 占位，但实际已创建了 `boss_bomber.tscn`、`boss_nachi.tscn`、`boss_fortress.tscn`。SpawnManager 通过 `_get_enemy_scene_path` 获取路径，而 LevelBase 的 `_spawn_boss` 使用硬编码路径 `"res://scenes/bosses/%s.tscn"`，两者不一致。

**建议**: 更新 `_enemy_scene_map` 中 BOSS 类型的映射指向正确的场景路径。

### ⚠️ 接口一致性问题

#### 1. BossBase 与 EnemyBase 信号连接缺失

**位置**: `enemy_base.gd:236-246`
```gdscript
func _on_area_entered(area: Node) -> void: ...
func _on_body_entered(body: Node) -> void: ...
```

**问题**: EnemyBase 定义了 `_on_area_entered`/`_on_body_entered` 回调但 `_ready()` 中未连接信号，导致敌机碰撞检测失效。BossBase 通过动态创建 Hitbox Area2D 解决了此问题，但 EnemyBase 仍未修复。

**建议**: 在 `EnemyBase._ready()` 末尾添加信号连接：
```gdscript
if has_node("Hitbox"):
    $Hitbox.area_entered.connect(_on_area_entered)
    $Hitbox.body_entered.connect(_on_body_entered)
```

#### 2. stage_config.json 命名不一致

**遗留问题**: stage_03 的 `bg_layers` 引用旧名 `bg_salween_*` 应改为 `bg_nujiang_*`（Devlog 已记录）。

## 六、性能优化建议

### 1. 弹幕发射使用对象池

**位置**: `boss_base.gd:609-628`
```gdscript
func _spawn_bullet(pos: Vector2, dir: Vector2, speed: float, damage: int = 1) -> void:
    var bullet: Node2D = bullet_scene.instantiate()
    ...
    get_parent().add_child(bullet)
```

**问题**: BOSS 弹幕每秒生成大量子弹，直接 `instantiate()` 效率低下，且未归还对象池。

**建议**: 改用 `PoolManager.get_object()` 获取子弹，并在子弹销毁时调用 `PoolManager.return_object()`。

### 2. 场景切换时清理对象池

**位置**: `level_base.gd:150-161`
```gdscript
func end_level() -> void:
    is_level_ended = true
    ...
    _goto_result_scene()
```

**问题**: 场景切换前未调用 `PoolManager.return_all_active()`，可能导致内存泄漏或残留对象。

**建议**: 在 `end_level()` 和 `force_end_level()` 中添加对象池清理调用。
